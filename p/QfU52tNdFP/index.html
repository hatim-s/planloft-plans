Automation Cleanup Pipeline v2 — Ground-Up Rewrite Design

Date: 2026-07-24 Author: @hatim (claude-code) Status: IMPLEMENTED Implemented: 2026-07-24 — v2-7 flipped the production entry points, deleted the legacy pipeline, generated the module README from describePipeline(), and closed the release taste gate. Supersedes: the 9-PR refactor stack #26005..#26051 (refactor/automation-cleanup-p*) and its packages/llm-tools/TODO.md Ground truth: origin/main @ d911ae9752ec; stack tip @ 4ed0c46c9e66; PR #26696 ported from its pinned head @ 3e44eed2b4ac (D-7)

1. Context and history

The automation cleanup pipeline in packages/llm-tools repairs, normalizes, and validates AI-authored workflow definitions before they reach the platform. On main today it is a synchronous ordered reducer of 20 plugins (src/tools/cleanUpAutomationConfig.ts), fed by an async prefetch facade that fetches builder resources, resolves dynamic schemas, and prefetches storage/SQL entity schemas. It is exposed through two MCP tools (cleanup_automation_config, batch_upsert_nodes) and throws five typed aggregate error families serialized verbatim onto the wire.

A July 2026 audit produced a 9-PR refactor stack (phases 0–7, decisions D1–D6/Q7–Q12 locked in TODO.md). That stack built genuinely good bones: a mutative draft runner with structural sharing and no-op identity, a per-plugin reporter, a shared validation-error kit with a single guard, one canonical pill grammar, a unified graph/edge-classification layer, and a consolidated schema flattener. It completed on 2026-07-14 and rebased onto main on 2026-07-16.

Main did not stop. Between the stack's base and today, main added storage pill record-path rewriting, SQL entity normalization + validation (a fifth error family), variable output-schema mirroring, recursive filter-conditions validation, storage filter-column validation, output-pill type resolution for strict dynamic inputs, arbitrary numeric pill indices, and the variable-resource validation exclusion — seven behavioral families the stack lacks entirely, plus one direct grammar regression (the stack's [0]-only pill index). PR #26696 (call-automation target-schema resolution, ~3.6k insertions across 55 files) is in flight on the old layout. A third rebase would be a semantic merge across every hot seam the stack restructured, with rename/delete conflicts hiding behavior loss. A rewrite beats a third rebase; the reasoning is recorded in the review history and stands.

This spec defines v2 after design review. The review ratified the five-step taxonomy and shared-utility strategy, and ordered six structural changes now baked in: one draft and one collector for the whole run (§3.2), no cross-substep short-circuit — a precedence-preserving finalize decides the thrown error (§3.3), a redesigned unified violation vocabulary (error shapes are not frozen contracts; only the MCP envelope's top-level keys are, §6), substeps re-cut along validation-category/resource lines with authorized plugin merges (§3.1), shared utils as encapsulated categories leaning toward classes (§4), and the whole v2 stack built before anything merges, porting #26696 from its pinned head (§8). The old stack's characterization tests and the still-binding locked decisions (§2, §7) remain the safety net for everything except the deliberately redesigned error interior.

2. Behavioral ground truth

The surface being preserved is everything observable from the four entry points: cleanUpAutomationConfig (sync), cleanUpAutomationConfigWithSchemaValidation (async), applyUpdateWorkflow/batch_upsert_nodes, and the MCP tool handlers in registerWorkflowTools.ts → mcp.ts.

The v2 behavior contract in one sentence: for every input, v2 makes the same accept/reject decision as main-plus-#26696, throws an error whose winning family matches what today's sequential ordering would have thrown first, and produces the same cleaned workflow on acceptance — while the error payload interior is redesigned (§6) and additional findings from families that today never got to run are attached, never promoted.

3. Target architecture

3.1 Step and substep taxonomy

The five steps are ratified (D-1): normalize → resolveContracts → buildGraph → validateOutputs → validateInputs. Substeps are re-cut along validation-category / node-resource lines instead of one-substep-per-legacy-plugin, with two authorized merges. Every merge preserves the union of transforms in exact relative order; characterization tests keep passing with import/name updates only.

3.2 Single draft, single collector — the substep contract

One mutative.create() wraps the entire pipeline run. One ViolationsCollector instance lives for the whole run. There is no per-substep draft machinery and no per-substep collector: substeps receive the shared draft and a scoped reporter handle; attribution lives in the collector, which tags every entry with {stepId, substepId, family}. Findings that embed draft-backed values are detached once, in a single pass at finalize (the last operation inside the draft callback, before mutative revokes proxies) — not per-substep.

Whole-pipeline no-op identity (D-9) holds because of the single create(): a run in which no substep mutates returns the caller's exact object reference, mutative's native contract, with structural sharing on partial change.

export type AutomationSubstepKind = 'transform' | 'validate' | 'transformAndValidate';

export interface ScopedReporter {
  violation(entry: AutomationViolationInput): void;  // collector adds stepId/substepId/family
  note(entry: AutomationNoteInput): void;
}

export interface AutomationSubstepSpec<TReads extends AutomationPipelineContextKey> {
  readonly id: AutomationSubstepId;
  readonly legacyPluginNames: readonly AutomationConfigPluginName[]; // skip-compat + phase toggles
  readonly description: string;
  readonly kind: AutomationSubstepKind;
  /** Present ⇒ this substep's violations belong to a finding family (validators + schemaGenerator). */
  readonly family?: AutomationViolationFamily;
  /** Prerequisites: if any listed substep recorded blocking violations or threw, this substep is SKIPPED. */
  readonly requires?: readonly AutomationSubstepId[];
  readonly reads: readonly TReads[];
  run(
    draft: Draft<AutomationWorkflowDefinition>,
    context: Readonly<Pick<AutomationPipelineContext, TReads>>,
    reporter: ScopedReporter,
  ): void;
}
  • defineAutomationSubstep (const-generic factory) makes reads compile-time meaningful; the heterogeneous manifest uses one private type-erased adapter inside pipeline/; erasure never escapes.
  • Validation substeps must not write the draft — enforced by contract tests that run each validator in an isolated test harness draft and assert identity; the production engine trusts the contract.
  • Transforms report nothing unless family is declared (schemaGeneratorJsonSchema and generateGroupIds are the two mixed substeps; the single interface absorbs them without a third shape).
  • Because one draft spans the run, a substep that throws mid-mutation leaves its partial writes in the draft. This is acceptable only because the run is already doomed: the exception is captured as a blocking finding (§3.3), dependents with requires skip, and finalize throws — the partially mutated workflow is never published.

AutomationPipelineContext is a plain readonly object (no getter factory): resourceNodeTypeMap, nodeSchemaResources, entityTypeSchemas (undefined = prefetch absent, per I12), nodeDynamicSchemas, inputSchemaValidation, plus #26696's callAutomationTargetSchemas, callAutomationNodeTargets, callAutomationResolutionCompleted, plus the pipeline-owned outputPillOccurrencesByNode artifact (§3.4, D-5).

3.3 Execution engine — no short-circuit, precedence-preserving finalize

One immutable manifest; ordering lives nowhere else:

export const AUTOMATION_CLEANUP_PIPELINE = {
  name: 'automation-cleanup',
  steps: [normalizeStep, resolveContractsStep, buildGraphStep, validateOutputsStep, validateInputsStep],
} as const satisfies AutomationPipelineDefinition;

Runner semantics:

  • Serial, declaration order, inside one draft. Substeps cannot call next(), insert siblings, or mutate the registry. Step boundaries grant artifact lifetime and trace grouping, never parallelism.
  • A substep failure never aborts the run. Blocking violations do not stop execution; unexpected exceptions do not propagate. Every substep executes unless a declared prerequisite failed.
  • requires / SKIPPED. When any prerequisite in requires recorded blocking violations or threw, the dependent substep does not execute; the collector records {substepId, status:'skipped', reason, failedPrerequisite}. The declared prerequisite edges, exhaustively:
    • generateGroupIds requires validateTopology — group traversal on invalid topology is unsafe/corrupting (the known case, main parity: group IDs never ran after a topology throw).
    • validateOutputPills requires repairPills + inferNodeTypes — the occurrence artifact and producer map are only meaningful over repaired pills and typed producers (scout 4's stale-cache finding; scout 1 §4.2 producer rungs).
    • validateSqlEntityPaths requires normalizeSqlEntityPaths — it re-analyzes normalized SQL; running it on raw SQL manufactures UNKNOWN_ENTITY noise.
    • validateTopology requires inferNodeTypes + cleanAppOnlyContext — topology dispatches on node type and the app-only context rule; these matter only when a prerequisite threw (transforms record no violations), preventing false topology findings from a crashed normalizer.
    • Nothing else declares requires. validateDynamicOutputSchemas → validateOutputPills is deliberately not an edge: pill validation fails open on a missing dynamic schema (scout 1 §4.3), so it runs and its findings attach; finalize precedence still throws the dynamic-output family first, exactly as today.
  • Unexpected exceptions are findings. The runner catches a substep throw and records a blocking internal-family finding {code:'SUBSTEP_EXECUTION_FAILED', stepId, substepId, message: <original message>, details:{cause}} (attribution per D-8: raw message preserved, attribution in fields). Execution continues per requires.
  • Finalize decides the thrown error. After the last substep (and the single detachment pass), finalize picks the earliest blocking event by manifest position — which reproduces exactly the family today's sequential ordering would have thrown first: topology → dynamic-output → output-pills → input-schema (schemaGenerator) → sql → input-schema (final). An internal finding takes its substep's position. The thrown AutomationCleanupError carries the winning substep's violations as violations and all other collected findings, notes, and skip records in the payload (§6). If the winner is an internal finding, finalize instead rethrows the original error wrapped once (stepId/substepId/cause as fields, message unchanged) so MCP's TOOL_CALL_ERROR path behaves as today; the ledger rides on the wrapper's fields, not the wire.
  • No blocking events: the draft closes, identity contract applies (D-9), per-family notes-only console.warn policies fire (I8), the workflow returns.
  • Skip semantics. pluginsToSkip survives as a compatibility alias resolving to (substep, phase) per the §3.1 table — unchanged behavior for every existing caller and test, including silent-ignore of unknown names in the compat wrapper (characterized). Internal stepsToSkip expands to descendant substeps; neither is MCP-exposed (I17). resolvePipelineSelection() owns validation, expansion, and dependency warnings, consumed by runner and describePipeline().
  • Attribution & diagnostics. The collector's entries are the attribution record; the runner additionally yields opt-in SubstepExecutionRecord { stepId, substepId, status: 'completed'|'skipped'|'failed', durationMs, violationCount, noteCount } with an injectable clock (per-substep changed is gone — a single draft makes per-substep identity deltas unobservable; whole-run identity is the contract). Nothing is appended to MCP payloads.
  • Introspection. describePipeline() returns JSON-safe { steps: [{ id, description, substeps: [{ id, legacyPluginNames, description, kind, family?, requires?, reads }] }] } without executing; a snapshot test pins it and the README pipeline table is generated from it. mode: 'inspect' runs fully (it always could now — nothing short-circuits), returns the would-be finalize decision plus the complete ledger instead of throwing.

3.4 Pipeline artifacts

Two cross-substep artifacts with explicit owners, both step-scoped, both born inside the draft and dead before finalize:

  • WorkflowGraph (§4.1) — constructed by the buildGraph step runner from draft state after disableLoopCaptureIterations, injected only into validateTopology/generateGroupIds, dead at step end. Safe across the group-ID write because group IDs change no adjacency input. Never escapes the draft callback.
  • outputPillOccurrencesByNode — collected once from draft state at the end of the normalize step (after repairPills/normalizeStoragePills have finished rewriting pills), keyed by stable node index, consumed by validateOutputs. This fixes the stack's stale pre-normalization cache and its duplicate-node-ID aliasing while keeping the single-walk win. Prefetch-time discovery is a separate conservative scan that normalizes braces on a throwaway copy first (D-5).

3.5 Prefetch (async facade)

A planner class plus two functions (§4.9):

  1. PrefetchPlanner — pure, deterministic, network-free. fromWorkflow(workflow) accumulates deduplicated builder-resource keys (consumer pass + referenced-producer pill pass over the brace-normalized throwaway copy, D-5), storage entity-type candidates, SQL entity candidates, call-automation target IDs (#26696). extendWithBuilderResources(resources) is the necessary second pass — dynamic-loader needs are only knowable after builder resources arrive. plan() returns the readonly plan.
  2. executePrefetchPlan(plan, dependencies) — owns all I/O and concurrency (Promise.all waves as today), preserves the fail-closed/fail-open asymmetry (I13) and the three-state outcome semantics (I12). fetchEntityType and fetchEntityTypeOutcome stay distinct; their not-found contracts differ.
  3. createCleanupContext(plan, result, options) — pure readonly assembly with visible defaults; computed maps override same-named caller options as today.

The sync no-host path builds the same context shape with absent maps — the offline profile is the same code path with explicit unavailability, not a fork.

4. Shared utilities — encapsulated categories

Per the review directive, every util category is an explicit encapsulation layer with one file-or-folder each, leaning toward classes. A class must own lifecycle, cached derived state, or an invariant-bearing boundary; where a class is genuinely worse, the category keeps function modules but still gets the one-folder encapsulation treatment, with the reason stated in one line.

4.1 WorkflowGraph — class, folder graph/

Encapsulates: all graph derivation and edge policy — the maps built independently by topology and group IDs (4 consumers), the 5 duplicated edge-policy copies, the 2 join wrappers. Owns one workflow revision with lazily cached nodeMap/incoming/outgoing/root, delegating to the single edgeClassification.ts rule table (strategy map, §5.3). Step-scoped only; no mutation methods; multi-root/unreachable diagnostics exposed explicitly instead of the silent first-root pick.

export class WorkflowGraph {
  constructor(workflow: Readonly<AutomationWorkflowDefinition>);
  node(nodeId: string): AutomationNode | undefined;
  nodes(): readonly AutomationNode[];
  incoming(nodeId: string): readonly AutomationWorkflowEdge[];
  outgoing(nodeId: string): readonly AutomationWorkflowEdge[];
  root(): AutomationNode | undefined;      // first in-degree-zero (I14); roots() exposes cardinality
  classifyOutgoing(nodeId: string): ClassifiedEdges;
  findJoin(branchEdges: readonly AutomationWorkflowEdge[], ownerId: string): string | undefined;
}

4.2 ViolationsCollector — class, folder shared/violations/ (pipeline-scoped)

Encapsulates: the whole run's finding ledger — every violation, note, skip record, and captured exception, tagged with {stepId, substepId, family}. The runner constructs one per run; substeps see only the narrow ScopedReporter view from scoped(stepId, substepId, family), so leaf code can neither clear nor inspect partial findings.

export class ViolationsCollector {
  scoped(stepId: AutomationPipelineStepId, substepId: AutomationSubstepId,
         family?: AutomationViolationFamily): ScopedReporter;
  recordSkip(substepId: AutomationSubstepId, failedPrerequisite: AutomationSubstepId): void;
  recordException(stepId: AutomationPipelineStepId, substepId: AutomationSubstepId, error: unknown): void;
  hasBlocking(substepId: AutomationSubstepId): boolean;   // drives `requires`
  finalize(): FinalizedFindings;   // ONE detachment pass; one-shot; called inside the draft tail
}

Class justified: one-shot lifecycle, the detach-at-finalize invariant, and the blocking index that requires evaluation reads.

4.3 SchemaIndex — class, file validation/inputSchema/SchemaIndex.ts

Encapsulates: repeated path lookups and direct-child scans across the eight input rules and the output-pill flatten caches (9 flatten call sites). Thin wrapper over FlattenedSchemaResult with lazy directChildKeys caching. The flatten algorithm stays a function (flattenSchema.ts) because it carries the bundling-sensitive retrieveSchemaInternal deep import (I16) and adapter injection; the class only indexes its result.

4.4 PillGrammar — encapsulated namespace-folder shared/pillGrammar/, internals stay functions

Encapsulates: the worst duplication in the package — 13 modules defining or consuming pill syntax. The folder is the only place regexes for pill syntax may live (grammar.ts is folder-private by convention and lint-guarded by an architecture test); scan.ts owns the offset-aware nested scanner; parse.ts owns parseOutputPillBody + format/normalize helpers and the PillExpression brand; policies.ts names the three semantic policies — anyTemplateExpression, recoverableBraceForm, validOutputPill — so consumers select semantics explicitly. Grammar content is main's (arbitrary [n], \S+ producer IDs); the agreement-table test updates accordingly. Class rejected in one line: callers parse once and read fields — an instance per occurrence buys allocation, not an invariant; the folder boundary supplies the encapsulation a class would.

4.5 SimilarityScorer — class, folder shared/similarity/

Encapsulates: the 2 byte-identical Levenshtein copies + 2 Fuse constructions. Now a class because it finally has state worth owning: one instance holds the normalized-candidate cache and a reused Fuse index per candidate set, fixing the per-call single-item Fuse instantiation the old stack claimed but never completed (scout 2 §3.9). normalizeIdentifier/getEditDistance remain pure module functions in the same folder; domain thresholds/ranking stay outside.

4.6 resourceIdentity — function module + brand, file shared/resourceIdentity.ts

Encapsulates: 17 extraction sites + 4 direct `${app}::${resource}` formattings behind getNodeResourceIdentity(node) → {appName, resourceName, key} and the ResourceKey brand, minted only here. Class worse in one line: stateless parse/format with no lifecycle — a brand plus two named functions is the entire invariant.

4.7 notesPolicy — function module, file shared/notesPolicy.ts

Encapsulates: the 4 policy call sites and the two intentionally different defaults (I8: input ← WORKFLOW_CLEANUP_VERBOSE, output ← NODE_ENV==='development') behind shouldCollectValidationNotes(family, options) so "cleanup" can never merge them. Class worse in one line: two pure predicates over env/options; instantiating policy would be ceremony.

4.8 json — function modules, folder shared/json/

Encapsulates: the 9 hand-rolled deep walkers (5 on main — the pattern re-grew after the stack removed 4) and the 10 subtly disagreeing isRecord guards. mapJsonValue (identity-preserving), foldJsonValue (path-aware, read-only), jsonRecord (one prototype-checked predicate). Arrays/plain records only; storage-pill rewriting becomes single-pass. Class worse in one line: pure structural recursion with no state — the folder is the boundary.

4.9 PrefetchPlanner — class, folder prefetch/

Encapsulates: the 9 repeated full-workflow discovery scans behind genuinely stateful two-pass accumulation (§3.5): builder-resource keys are known up front, dynamic-loader needs only after resources arrive, so the instance's lifecycle is the two-phase protocol — fromWorkflow() → extendWithBuilderResources() → plan(). It never fetches; executePrefetchPlan and createCleanupContext remain pure-boundary functions because I/O ownership and readonly assembly have one-way data flow with no invariant between calls.

The old validationErrorKit category dissolves: with one error class and one guard (§6), there is nothing left to generate — shared/violations/ owns the entire vocabulary.

5. Design patterns charter

Distilled from the conventions scout into enforceable rules; each has in-repo precedent. Repo law (200-line files, no barrels, no single-letter iterators, named exports, interfaces-over-types, const maps not enums, satisfies, .js ESM specifiers, no /src imports, 100-col format) applies to every file and is not repeated per rule.

  1. One declarative manifest, two levels max. Steps contain substeps; array order is execution order; entries carry only id + direct function reference + family/requires metadata. No glob discovery, DI, priorities, dependency sorting beyond declared requires, string lookup, or inline lambdas. Ordering rationale is a one-line comment above the entry: <Verb> <fact> before <consumer> reads <dependent fact>.
  2. Greppability ceiling. Grep one substep ID → manifest entry, implementation, tests, changelog. If reading the manifest requires decoding predicates or factory-generated handlers, it has become a framework: simplify.
  3. Strategy map when one discriminator drives ≥2 behaviors; exhaustive switch for one local operation on a discriminated union. Never two switches on the same discriminator. Precedent: edgeClassification.ts rule table; main's two-variant switch (loader.operationId) staying a switch.
  4. Branded strings only where a real grammar is repeatedly parsed/formatted: PillExpression, ResourceKey, GroupId. Only named parse/format/create functions mint them; convert to plain strings at generated-model/JSON/MCP boundaries; never brand ordinary IDs. No runtime value-object classes for strings.
  5. Findings are data; nothing inside the pipeline throws. Validators report through the scoped reporter and never throw for workflow content; the runner captures unexpected substep exceptions as internal-family findings; finalize is the only throw site. Fail-open unavailability is a note or skip, never a throw. Traversal control uses explicit continue/skipSubtree, never exceptions.
  6. Error vocabulary grammar. One violation shape (§6); family names lower-camel in code, envelope codes AUTOMATION_<TOPIC>_VALIDATION_FAILED, finding codes concise upper-snake conditions (MALFORMED_PILL) without the envelope prefix. shared/violations/ owns representation only — never throw/log/suppress policy.
  7. Immutability contract. Public entry points never mutate input. Exactly one mutative.create() per pipeline run; only substep adapters see Draft; parsers/planners/rankers/graph-builders/validators are draft-free pure functions; drafts never cross await, caches, contexts, or error payloads (the collector's single finalize pass detaches before the draft closes); a fully no-op run preserves the caller's reference. Local Map/Set accumulation inside a pure-result function is fine when the accumulator does not escape mutable.
  8. Two traversal algebras, no universal Visitor. Path-aware JSON fold + identity-preserving JSON map for payloads; explicit for...of over nodes/edges in the owning step; named BFS/DFS graph helpers for control flow. Domain parsers consume leaf values; they do not own recursion.
  9. Naming. Files and executable IDs lower-camel verb-first sharing one stem (validateTopology = id = function = file); leaf checks are check<Invariant>; ordered polymorphic objects are <subject>ValidationRule; tests <subject>.test.ts / <substep>.pipeline.test.ts. Ban util, helper, manager, process, data as name substance.
  10. Documentation placement. Code comment = present-tense invariant; manifest comment = ordering dependency; JSDoc = non-obvious contract (identity, fail-open, wire shape, the RJSF deep-import constraint) only; one root automation/README.md with a describePipeline()-generated order section; named specs (OUTPUT_PILL_VALIDATION.md) only for genuine grammars/protocols; changelog = trigger/tried/decision history.
  11. No i18n in this package — MCP/backend diagnostics, sentence case, wire-visible text governed by §6 (not byte-frozen; key-stable).
  12. Release taste gate: the conventions scout's 12-question rubric (≥20/24, zeros on questions 2/6/7/8/12 blocking) runs at the flip PR review.

6. Error and reporting contract

Design freedom, one constraint. Error codes, class definitions, and payload shapes are not frozen contracts — the five copy-pasted legacy families are replaced by one unified vocabulary and one error class. The single constraint: the MCP envelope's top-level keys code, message, violations, notes remain present so envelope-parsing consumers don't break. New keys may be added; interior shapes are redesigned.

Unified violation shape — one shape across all families:

export type AutomationViolationFamily =
  | 'topology' | 'dynamicOutput' | 'outputPill' | 'inputSchema' | 'sql'
  | 'groupId'      // advisory: silent group-ID failures become first-class findings (D-3)
  | 'internal';    // captured unexpected substep exceptions

export interface AutomationViolation {
  readonly family: AutomationViolationFamily;
  readonly code: string;           // per-family const map, e.g. MALFORMED_PILL, MISSING_ROOT
  readonly stepId: AutomationPipelineStepId;
  readonly substepId: AutomationSubstepId;
  readonly nodeId?: string;
  readonly path?: string;          // input path / pill path / schema pointer, family-normalized
  readonly message: string;
  readonly details?: JsonRecord;   // suggestions, expected/actual, targetAutomationId (#26696), cause summary
}
// AutomationNote: same shape; notes never block (I8).
  • Blocking families: topology, dynamicOutput, outputPill, inputSchema, sql, internal. groupId is advisory (D-3): MISSING_ROOT/CYCLE/CONFLICTING_ASSIGNMENT are recorded and surfaced in the ledger — real corruption is no longer silent — but never win finalize, preserving main's accept/reject behavior. Promoting the family to blocking is a one-line flip plus a decision entry, later.
  • One error class, one guard. AutomationCleanupError is built by finalize only. Envelope:
{
  code: 'AUTOMATION_<TOPIC>_VALIDATION_FAILED',   // winning family's envelope code (values kept per family)
  message: '<winning family message>',
  violations: AutomationViolation[],   // the winning substep's blocking violations (today's payload position)
  notes: AutomationNote[],             // full-run notes
  findings: AutomationViolation[],     // complete ledger: every family incl. groupId advisories + internal
  skipped: [{ substepId, failedPrerequisite, reason }],
}

isAutomationCleanupError is the single guard, consumed by exactly two call sites: finalize/wrapper dispatch and stringifyWorkflowToolError. A completeness test enumerates blocking families in the vocabulary and asserts finalize can produce each — new families cannot silently miss dispatch.

  • internal winner exception: when the earliest blocking event is a captured substep exception, finalize rethrows the original error wrapped once (stepId/substepId/cause as fields, public message unchanged — D-8); MCP serializes it as TOOL_CALL_ERROR exactly as unexpected errors serialize today. The ledger rides on wrapper fields, not the wire.
  • The groupId typed-error question (old OD-3) dissolves here: silent failures are first-class groupId-family violations in the same vocabulary — no separate error class, no separate wire family.
  • Notes policy stays per-family (I8) behind shouldCollectValidationNotes; finalize owns the per-family JSON console.warn for notes-only runs; leaf code never logs.
  • Dispatch layers: applyUpdateWorkflow keeps {success:false, error, violations, notes?, errorCause} populated from the new envelope; MCP keeps content:[{type:'text',text}] + isError:true; unrecognized failures keep {code:'TOOL_CALL_ERROR', tool, message, violations?, notes?}.
  • Wire statement: MCP tool input schemas (including the required connectionMap no-op field, D-2), success JSON, and warning mechanics are unchanged. Error envelopes are key-stable, shape-redesigned; v2-0 snapshots the new envelope as the contract going forward.
