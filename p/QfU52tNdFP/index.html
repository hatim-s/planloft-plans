Automation Cleanup Pipeline v2 — Ground-Up Rewrite Design

Date: 2026-07-24 Author: @hatim (claude-code) Status: Proposed — pending OPEN DECISIONS sign-off (§10) Supersedes: the 9-PR refactor stack #26005..#26051 (refactor/automation-cleanup-p*) and its packages/llm-tools/TODO.md Ground truth: origin/main @ d911ae9752ec; stack tip @ 4ed0c46c9e66; PR #26696 @ 3e44eed2b4ac

1. Context and history

The automation cleanup pipeline in packages/llm-tools repairs, normalizes, and validates AI-authored workflow definitions before they reach the platform. On main today it is a synchronous ordered reducer of 20 plugins (src/tools/cleanUpAutomationConfig.ts), fed by an async prefetch facade that fetches builder resources, resolves dynamic schemas, and prefetches storage/SQL entity schemas. It is exposed through two MCP tools (cleanup_automation_config, batch_upsert_nodes) and throws five typed aggregate error families serialized verbatim onto the wire.

A July 2026 audit produced a 9-PR refactor stack (phases 0–7, decisions D1–D6/Q7–Q12 locked in TODO.md). That stack built genuinely good bones: a mutative draft runner with structural sharing and no-op identity, a per-plugin reporter, a shared validation-error kit with a single guard, one canonical pill grammar, a unified graph/edge-classification layer, and a consolidated schema flattener. It completed on 2026-07-14 and rebased onto main on 2026-07-16.

Main did not stop. Between the stack's base and today, main added storage pill record-path rewriting, SQL entity normalization + validation (a fifth error family), variable output-schema mirroring, recursive filter-conditions validation, storage filter-column validation, output-pill type resolution for strict dynamic inputs, arbitrary numeric pill indices, and the variable-resource validation exclusion — seven behavioral families the stack lacks entirely, plus one direct grammar regression (the stack's [0]-only pill index). PR #26696 (call-automation target-schema resolution, ~3.6k insertions across 55 files) is in flight on the old layout. A third rebase would be a semantic merge across every hot seam the stack restructured — the registry, the prefetch wrapper, the context bag, and the input/output validation modules — with rename/delete conflicts hiding behavior loss.

A rewrite beats a third rebase for three reasons. First, the stack's own architecture shows strain it cannot fix incrementally: a flat 16-entry array whose dependency structure lives in comments, a context bag accreting untyped artifacts, validators forced through a mutating-draft interface, and a stale pill-occurrence cache that validates pre-normalization text. Second, the drift is not textual but semantic — porting seven feature families into a moved-and-renamed tree is riskier than porting proven foundations onto current main's behavior. Third, the user's target design (steps with substeps, typed shared utilities, explicit patterns) is a different shape than the stack's flat registry; rebasing would preserve the wrong skeleton.

This spec defines v2: same observable behavior as main-plus-#26696, new internal architecture. The old stack's characterization tests and still-binding locked decisions (§2, §7) remain the safety net. Behavior changes are never smuggled in — each one is an OPEN DECISION (§10) or an explicitly gated follow-up.

3. Target architecture

3.1 Step taxonomy — five dependency-oriented steps

Chosen: the pipeline-design scout's Candidate A. It is the smallest decomposition where step boundaries coincide with real dependency edges, and it maps all 20 substeps with zero reordering — the five-step partition falls exactly on main's existing sequence. Candidate B (three coarse phases) makes "Rewrite" a 12-substep catch-all that defeats context slicing; Candidate C (artifact stages including prefetch) couples the taxonomy rewrite to network/prefetch changes that need separate characterization — it remains a plausible later evolution, not the first move.

3.2 Substep contract

One mutative draft boundary and one fresh FindingCollector per substep. Context is a compile-time slice declared through reads; the reporter is a separate parameter so "data read" and "effects emitted" never blur. Validators receive the same signature but their tests assert draft identity — the kind field is enforced by contract tests, not by a second interface (two interfaces would force the two genuine mixed substeps, schemaGeneratorJsonSchema and generateGroupIds, into an awkward third shape).

export type AutomationSubstepKind = 'transform' | 'validate' | 'transformAndValidate';

export interface SubstepReporter<TViolation, TNote> {
  violation(entry: TViolation): void;
  note(entry: TNote): void;
}

export interface SubstepFindingPolicy<TViolation, TNote> {
  /** Present ⇒ violations block: the runner creates and throws this typed error. */
  readonly createError?: (
    violations: readonly TViolation[],
    notes: readonly TNote[],
  ) => Error;
  /** Present ⇒ notes-only runs emit this console.warn per the existing policies (I8). */
  readonly notesWarningMessage?: string;
}

export interface AutomationSubstepSpec<
  TReads extends AutomationPipelineContextKey,
  TViolation,
  TNote,
> {
  readonly id: AutomationConfigPluginName;   // stable — pluginsToSkip compat (I17)
  readonly description: string;
  readonly kind: AutomationSubstepKind;
  readonly reads: readonly TReads[];
  readonly findings?: SubstepFindingPolicy<TViolation, TNote>;
  run(
    draft: Draft<AutomationWorkflowDefinition>,
    context: Readonly<Pick<AutomationPipelineContext, TReads>>,
    reporter: SubstepReporter<TViolation, TNote>,
  ): void;
}

export function defineAutomationSubstep<
  const TReads extends AutomationPipelineContextKey,
  TViolation = never,
  TNote = never,
>(spec: AutomationSubstepSpec<TReads, TViolation, TNote>) {
  return spec;
}

export interface AutomationPipelineStep {
  readonly id: AutomationPipelineStepId;      // 'normalize' | 'resolveContracts' | …
  readonly description: string;
  readonly substeps: readonly AutomationSubstep[];  // runner-private erased form
}

The heterogeneous manifest needs one private type-erased adapter inside the runner; leaf modules keep their precise finding types. Erasure never escapes pipeline/.

AutomationPipelineContext is a plain readonly object (no getter factory): resourceNodeTypeMap, nodeSchemaResources, entityTypeSchemas (undefined = prefetch absent, per I12), nodeDynamicSchemas, inputSchemaValidation, plus #26696's callAutomationTargetSchemas, callAutomationNodeTargets, and callAutomationResolutionCompleted, plus the pipeline-owned outputPillOccurrencesByNode artifact (§3.4, OD-5).

3.3 Execution engine

One immutable manifest; ordering lives nowhere else:

export const AUTOMATION_CLEANUP_PIPELINE = {
  name: 'automation-cleanup',
  steps: [normalizeStep, resolveContractsStep, buildGraphStep, validateOutputsStep, validateInputsStep],
} as const satisfies AutomationPipelineDefinition;

Runner semantics:

  • Serial, declaration order. Substeps cannot call next(), insert siblings, or mutate the registry. Step boundaries grant cache lifetime and trace grouping, never parallelism.
  • One mutative.create() per substep. No-op substeps return the original reference; after 20 no-ops the public result is the caller's object (identity delta gated by OD-9). Validation substeps' drafts are asserted untouched by contract tests.
  • Skip semantics. pluginsToSkip survives as a compatibility alias resolving directly to substep IDs — unchanged behavior for every existing caller and test, including silent-ignore of unknown names in the compat wrapper (characterized). An internal stepsToSkip expands to descendant substep IDs before execution; neither is MCP-exposed (I17). A single resolvePipelineSelection() owns validation, expansion, and dependency warnings, consumed by both runner and describePipeline().
  • Per-substep attribution. Each run yields a SubstepExecutionRecord { stepId, substepId, status: 'completed'|'skipped'|'blocked'|'failed', changed, durationMs, violationCount, noteCount } with an injectable clock. Records are opt-in diagnostics; the default public result is only the workflow. Nothing is appended to MCP payloads.
  • Error aggregation policy, per step:
    • Within one validation substep: collect every violation in that domain, then the runner evaluates findings.createError immediately after the substep returns. normalize/resolveContracts substeps have no finding policy — they are fail-open transforms.
    • Across substeps: stop at the first blocking substep. buildGraph.validateTopology and validateOutputs.validateDynamicOutputSchemas are in-step barriers; group IDs never run on invalid topology, pill paths never validate against missing dynamic contracts. This preserves main's first-blocking-error precedence exactly.
    • Unexpected throws: fail immediately, wrapped once with stepId/substepId/cause. The public message stays the original error message for parity (adopting the stack's Automation cleanup plugin "<name>" failed: prefix is OD-8). Typed automation errors pass through unwrapped, recognized by the single guard (§6).
    • Notes: attach to the substep's typed error when violations exist; otherwise apply that substep's notesWarningMessage policy and continue (I8).
  • Introspection. describePipeline() returns JSON-safe { steps: [{ id, description, substeps: [{ id, description, kind, reads }] }] } without executing; a snapshot test pins it and the README pipeline table is generated from it — the mechanism that would have caught the stack's 16-vs-20 drift. mode: 'inspect' runs in memory, returns blockedBy instead of throwing, and marks later substeps blocked rather than executing them against rejected state. No "collect across all validators" mode: it would validate states the production order never produces.

3.4 Pipeline artifacts

The stack's context bag failed because artifacts had no provenance or freshness. v2 gives the two cross-substep artifacts explicit owners:

  • WorkflowGraph (§4.1) — constructed by the buildGraph step runner from the immutable result of disableLoopCaptureIterations, injected only into validateTopology/generateGroupIds, dead at step end. Safe across the group-ID write because group IDs change no adjacency input.
  • outputPillOccurrencesByNode — collected once from the immutable result of the normalize step (after fixPillMappings/triggerWebhookPillBody/storagePillRecordPaths have finished rewriting pills), keyed by stable node index, consumed by validateOutputs. This fixes the stack's stale pre-normalization cache and its duplicate-node-ID aliasing while keeping the single-walk win. Prefetch-time discovery is a separate conservative scan (OD-5 governs whether that scan also normalizes braces first).

3.5 Prefetch (async facade)

Three named phases replace the monolithic wrapper — functions, not a builder class (§4.6):

  1. buildPrefetchPlan(workflow) — pure, deterministic: deduplicated builder-resource keys (consumer pass + referenced-producer pill pass), storage entity-type candidates, SQL entity candidates, call-automation target IDs (#26696). Two-phase by necessity: dynamic-loader needs are only knowable after builder resources arrive, so the plan has an extendWithBuilderResources(resources) second pass.
  2. executePrefetchPlan(plan, dependencies) — owns all I/O and concurrency (Promise.all waves as today), preserves the fail-closed/fail-open asymmetry (I13) and the three-state outcome semantics (I12). fetchEntityType and fetchEntityTypeOutcome stay distinct; their not-found contracts differ.
  3. createCleanupContext(plan, result, options) — pure readonly assembly with visible defaults; computed maps override same-named caller options as today.

The sync no-host path builds the same context shape with absent maps — the offline profile is the same code path with explicit unavailability, not a fork.

4. Shared utilities

The user asked for util classes. The honest rule applied below: a class must own lifecycle, cached derived state, or an invariant-bearing boundary — otherwise it is a function. Three classes earn it; the rest are rejected with the reason stated.

4.1 WorkflowGraph — class ✅

Collapses: graph maps built independently by topology and group IDs in both trees (4 direct consumers), the duplicated edge-policy copies (5 sites: classifyOutgoingEdges, getBranchEdgeProps, topology edgeUtils, topology traversability, two join wrappers), and the 2 join-finding wrappers per tree. Owns one workflow revision with lazily cached nodeMap/incoming/outgoing/root, and delegates to the single edgeClassification.ts rule table (strategy map, §5) for classifyOutgoing/findJoin. Step-scoped only (§3.4); no mutation methods; multi-root/unreachable diagnostics exposed explicitly instead of the silent first-root pick. Class justified: related derived views with one lifetime and real lazy caching.

export class WorkflowGraph {
  constructor(workflow: Readonly<AutomationWorkflowDefinition>);
  node(nodeId: string): AutomationNode | undefined;
  nodes(): readonly AutomationNode[];
  incoming(nodeId: string): readonly AutomationWorkflowEdge[];
  outgoing(nodeId: string): readonly AutomationWorkflowEdge[];
  root(): AutomationNode | undefined;      // first in-degree-zero (I14); roots() exposes cardinality
  classifyOutgoing(nodeId: string): ClassifiedEdges;
  findJoin(branchEdges: readonly AutomationWorkflowEdge[], ownerId: string): string | undefined;
}

4.2 FindingCollector — class ✅

Collapses: the stack's closure-based reporter plus main's five in-plugin aggregation patterns. Owns the two arrays, draft detachment before mutative revokes proxies (the one subtle lifecycle behavior that justifies a class), and one-shot snapshot() returning detached readonly arrays. The runner constructs one per substep and injects only the narrow SubstepReporter view, so leaf code can neither clear nor inspect partial findings. ≤80 lines.

4.3 SchemaIndex — class ✅

Collapses: repeated path lookups and direct-child scans across the eight input rules and the output-pill flatten caches (9 direct flatten call sites across both trees). Thin wrapper over FlattenedSchemaResult with lazy directChildKeys caching — the hoisting the old stack's Phase 5 wanted. The flatten algorithm itself stays a function (flattenSchema.ts) because it carries the bundling-sensitive retrieveSchemaInternal deep import (I16) and adapter injection; the class only indexes its result.

4.4 pillGrammar — functions + brand, class ❌

Collapses the worst duplication in the package: 13 modules defining or consuming pill syntax (loose fixer pattern, webhook derivative, strict parser, two generic template regexes, storage brace test, formula }}-seeker, dynamic-schema template regex, …). One module owns the offset-aware nested scanner, parseOutputPillBody, format/normalize helpers, and three named policies — anyTemplateExpression, recoverableBraceForm, validOutputPill — so consumers select semantics explicitly instead of re-deriving regexes. Grammar content is main's (arbitrary [n] indices, \S+ producer IDs); the agreement-table test is updated accordingly. A PillExpression string brand (§5) marks parsed output. Class rejected: callers parse once and read fields — an instance per occurrence buys allocation, not an invariant.

4.5 validationErrorKit + single guard — factory functions, class ❌ (it generates classes)

Collapses: 5 copy-pasted error envelopes on main (a 6th if OD-3 adds group IDs) plus the duplicated guard list in workflowToolErrors.ts — 10 envelope definitions across the union. createValidationErrorKit({ name, errorCode, message, includeNotes }) produces class + factory + guard with exact legacy JSON (the dynamic-output family keeps its no-notes envelope). isAutomationValidationError is the one union guard consumed by both the runner pass-through and MCP serialization; a table-completeness test proves every blocking substep's error family is registered — closing the stack's "manually extended union" gap.

4.6 Prefetch — three pure functions, builder class ❌

The pipeline-design scout floated PrefetchPlanBuilder; the conventions scout rejected fluent builders and this spec sides with the latter: the three phases (§3.5) have natural one-way data flow, there is no invariant to protect between .add() calls, and pure functions are independently unit-testable (taste test 9). Rejected in one sentence: a builder's only state would be accumulation that a plain function's local Map already provides. Collapses the 9 repeated full-workflow discovery scans into one plan pass plus the necessary post-fetch extension.

5. Design patterns charter

Distilled from the conventions scout into enforceable rules; each has in-repo precedent. Repo law (200-line files, no barrels, no single-letter iterators, named exports, interfaces-over-types, const maps not enums, satisfies, .js ESM specifiers, no /src imports, 100-col format) applies to every file and is not repeated per rule.

  1. One declarative manifest, two levels max. Steps contain substeps; array order is execution order; entries carry only id + direct function reference + finding policy. No glob discovery, DI, priorities, dependency sorting, string lookup, or inline lambdas. Ordering rationale is a one-line comment above the entry in the grammar <Verb> <fact> before <consumer> reads <dependent fact>. Precedent: main's ordered registry + the page pipeline's visible macro stages (cleanUpConfig.ts:462-520); stack README declaring order observable.
  2. Greppability ceiling. Grep one substep ID → manifest entry, implementation, tests, changelog. If reading the manifest requires decoding predicates or factory-generated handlers, it has become a framework: simplify.
  3. Strategy map when one discriminator drives ≥2 behaviors; exhaustive switch for one local operation on a discriminated union. Never two switches on the same discriminator. Precedent: stack's edgeClassification.ts rule table replacing main's parallel switches; main's two-variant switch (loader.operationId) staying a switch.
  4. Branded strings only where a real grammar is repeatedly parsed/formatted: PillExpression, ResourceKey, GroupId. Only named parse/format/create functions mint them; convert to plain strings at generated-model/JSON/MCP boundaries; never brand ordinary IDs. No runtime value-object classes.
  5. Findings are data; exceptions mean the substep could not execute its contract. Validators report through the injected reporter and never throw for workflow content; the runner alone converts blocking findings to typed errors (policy in the manifest, representation in the kit). Fail-open unavailability is a note or skip, never a throw. Traversal control uses explicit continue/skipSubtree, never exceptions. Precedent: page violation checkers (pure, list-returning) + stack reporter/runner split.
  6. Error vocabulary grammar. Automation<Topic>ValidationError / create… / is… / aggregate isAutomationValidationError; envelope codes AUTOMATION_<TOPIC>_VALIDATION_FAILED; finding codes concise upper-snake conditions (MALFORMED_PILL) without the envelope prefix. The kit owns representation only — never throw/log/suppress policy.
  7. Immutability contract. Public entry points never mutate input. Exactly one mutative.create() per substep; only the thin adapter sees Draft; parsers/planners/rankers/graph-builders/validators are draft-free pure functions; drafts never cross await, caches, contexts, or error payloads; no-op preserves the original reference. Local Map/Set accumulation inside a pure-result function is fine when the accumulator does not escape mutable.
  8. Two traversal algebras, no universal Visitor. Path-aware JSON fold + identity-preserving JSON map for payloads; explicit for...of over nodes/edges in the owning step; named BFS/DFS graph helpers for control flow. Domain parsers consume leaf values; they do not own recursion.
  9. Naming. Files and executable IDs lower-camel verb-first sharing one stem (validateTopology = id = function = file); only draft adapters get the Plugin suffix; leaf checks are check<Invariant>; ordered polymorphic objects are <subject>ValidationRule; tests <subject>.test.ts / <substep>.pipeline.test.ts. Ban util, helper, manager, process, data as name substance.
  10. Documentation placement. Code comment = present-tense invariant; manifest comment = ordering dependency; JSDoc = non-obvious contract (identity, fail-open, wire shape, the RJSF deep-import constraint) only; one root automation/README.md with a describePipeline()-generated order section; named specs (OUTPUT_PILL_VALIDATION.md) only for genuine grammars/protocols; changelog = trigger/tried/decision history. Never a runtime invariant only in a changelog; never PR history in a code comment.

6. Error and reporting contract

  • Typed families (wire-frozen): AUTOMATION_INPUT_SCHEMA_VALIDATION_FAILED, AUTOMATION_OUTPUT_PILL_VALIDATION_FAILED, AUTOMATION_DYNAMIC_OUTPUT_SCHEMA_VALIDATION_FAILED (no notes field — historical shape preserved), AUTOMATION_TOPOLOGY_VALIDATION_FAILED, AUTOMATION_SQL_VALIDATION_FAILED.   • Single guard. isAutomationValidationError is consumed by exactly two call sites: runner pass-through (typed errors skip attribution wrapping) and stringifyWorkflowToolError.   • Unexpected errors wrap once: { stepId, substepId, cause }, public message unchanged pending OD-8.
  • Notes policy stays per-family (I8) behind shouldCollectValidationNotes; the runner owns the single JSON console.warn; leaf code never logs.
  • Dispatch layers unchanged: applyUpdateWorkflow keeps {success:false, error, violations, notes?, errorCause}; MCP keeps content:[{type:'text',text}] + isError:true; unrecognized failures keep {code:'TOOL_CALL_ERROR', tool, message, violations?, notes?}.

10. OPEN DECISIONS

Each lists the recommendation first. Everything not listed here is decided by this spec.

OD-1 — Step taxonomy

Recommend: Candidate A — five dependency-oriented steps (normalize → resolveContracts → buildGraph → validateOutputs → validateInputs), zero reordering, all 20 names as stable substep IDs. Alt B: three coarse barrier phases — smaller engine, but a 12-substep "Rewrite" catch-all defeats context slicing and step-scoped caches. Alt C: artifact-producing stages incl. prefetch — best introspection, but couples the rewrite to network-semantics changes; viable later evolution on top of A.

OD-2 — connectionMap MCP field

Recommend: keep it required and no-op with the existing backward-compat description; delete only the internal machinery (dormant plugin, context field, unused options) in v2-8. Alt: make it optional + deprecated — small schema change, needs caller inventory. Alt: delete it (the old stack's choice) — a breaking MCP contract change that does not belong in a structural rewrite.

OD-3 — Group-ID failure semantics

Recommend: main parity in the rewrite (silent no-op on missing root, silent stop on conflicting assignment), with the stack's typed AutomationGroupIdGenerationError (MISSING_ROOT / CYCLE / CONFLICTING_ASSIGNMENT) as an explicit follow-up PR after v2-8. The typed error is better engineering but is a new wire-visible error family — a behavior decision, not a rewrite. Alt: adopt the typed error inside v2-4 — one less PR, but mixes a wire change into a parity-gated slice. Alt: never adopt — leaves real corruption silent.

OD-4 — Join-traversal unification (old D2)

Recommend: adopt the unified type-filtered edge traversal for both topology and group IDs, gated on the four fixtures producing identical output (the old stack verified this and survived review, including the Loop/Scope Next-fallback fix). Main's group-ID path still hands all outgoing edges to FlowBuilder.findJoinNodeId, so malformed graphs can differ. Alt: preserve main's split semantics — pure parity, but permanently forks graph policy into two copies and re-inherits the drift the stack fixed. Alt: unify later — carries two edge policies through the whole stack.

OD-5 — Pill-occurrence freshness and the raw-brace prefetch gap

Recommend: collect occurrences once after the normalize step (fixing the stack's stale-cache bug, keyed by node index), and normalize braces in a throwaway copy before the generic prefetch producer scan — extending the fix #26696 made for call-automation only. Discovery becomes slightly broader (spaced-brace producers get their resources fetched); validation semantics are unchanged since validators already see post-fix text. Alt: preserve the gap byte-for-byte — pure parity, but leaves a known inconsistency where { { x } } producers are invisible to prefetch yet valid after repair. Alt: per-validator re-scan (main's model) — correct but rewalks the workflow ~6 times.

OD-6 — FlowBuilder dependency direction

Recommend: extract the pure findJoinNodeId/getGroupId kernel plus flow enums to packages/defs as v2-1 (the old stack already proved the enum half; platform keeps compat re-exports). A package importing apps/platform UI code is an inverted dependency and blocks the architecture test. Alt: keep importing platform FlowBuilder — zero risk now, entrenches the inversion. Alt: copy the algorithms into llm-tools — forks group-ID semantics from the UI, the one thing I14 forbids.

OD-7 — PR #26696 absorption

Recommend: land #26696 on main first, then rebuild v2 on the merged baseline using its 72 test declarations as the parity oracle in v2-5/v2-6. Alt: port from the PR head into v2 and close the PR — decouples from its review timeline but risks divergence if the PR evolves. Alt: ask for a rebase onto v2 — repeats the stale-stack failure mode on someone else's branch.

OD-8 — Unexpected-error public message

Recommend: keep the raw error message (main parity) with stepId/substepId/cause as internal fields; adopt the stack's Automation cleanup plugin "<name>" failed: … prefix as a tiny opt-in follow-up with its own error-contract test. Alt: adopt the prefix in v2-2 — better diagnostics day one, but an observable message change inside the rewrite. Alt: prefix only in MCP TOOL_CALL_ERROR serialization — splits the contract in two places.

OD-9 — Whole-pipeline no-op identity

Recommend: adopt the stack's identity contract — a fully no-op run returns the caller's exact object reference — as an explicit, tested delta in v2-4. Main's generateGroupIds always clones when a root exists, so this is a reference-only observable change (JSON output identical); callers using identity to detect change get a strict improvement. Alt: preserve main's cloning — parity, but forfeits the cheapest change-detection signal and the structural-sharing test suite's strongest assertion.
