Bolt Derived Path Dependencies Plan

Goal: add optional, generic, path-level derived state to Bolt without hardcoding component dependency semantics.

Target repo: /Users/admin/Projects/bolt

Primary files: src/bolt/bolt.ts, src/bolt/types.ts, src/bolt/utils.ts, README.md, new tests under src/bolt/.

Intent

<h3>Why We Are Doing This</h3>
<p>Bolt already solves path-indexed subscriptions: components read a path, writes notify that path and parent paths, and unrelated subscribers stay asleep. Dynamic UI builders need one more primitive: path-indexed derivation. A component's persisted state may be computed from other arbitrary paths, and downstream components may subscribe to that computed state as if it were normal store state.</p>
<ul>
  <li>Dynamic UI use case: any node/component can expose state at a path.</li>
  <li>Dependency use case: any node/component can derive state from any other path or set of paths.</li>
  <li>Chaining use case: <code>A.input</code> changes, <code>B.value</code> recomputes, then <code>C.value</code> recomputes from <code>B.value</code>.</li>
  <li>Mixed ownership use case: some paths stay manual; some paths are derived; some derived paths read manual override paths.</li>
  <li>Genericity requirement: Bolt must not know about components, forms, graphs, canvases, screens, or UI node IDs. It should only know paths, reads, writes, derivations, and listeners.</li>
</ul>

<h3>What We Are Doing</h3>
<p>Add a store-level <code>derive</code> API. A derived registration has one target path, a set of source paths, and a pure compute function. When a source path is affected by a write, Bolt recomputes the target path before notifying React subscribers. Derived writes flow through the same path notification system, so downstream subscribers and downstream derived targets update naturally.</p>
<ul>
  <li>New public primitive: <code>api.derive(targetPath, sourcePaths, compute, options?)</code>.</li>
  <li>New internal graph: derived nodes indexed by target path and source path.</li>
  <li>New update pipeline: user write -> derived graph settles -> deduped listener notification.</li>
  <li>New safety layer: cycle detection, duplicate target rejection, manual-write policy, dispose cleanup.</li>
  <li>New docs: explain derived paths as optional generic path runtime, not component graph feature.</li>
</ul>

<h3>How We Are Doing It</h3>
<p>Keep the core store path-centric. Register derived dependencies by normalized path keys. On every successful write, compute the transitive set of affected derived nodes, topologically order them, recompute targets, add changed targets back into the changed-path set, then notify all affected listeners once. If no derived registrations exist, the hot path stays almost identical to current Bolt.</p>
<ul>
  <li>Do not add React-only state. Derivation lives in <code>createInternalBoltStore</code>.</li>
  <li>Do not add selectors. Computation is explicit and target-path materialized.</li>
  <li>Do not add async derivation in v1. Async belongs in effects/jobs later.</li>
  <li>Do not allow arbitrary writes inside compute. Compute returns the target value only.</li>
  <li>Do not make derived state mandatory. Empty graph means current behavior.</li>
</ul>

Current Bolt Facts

  • createBoltStore(initialState) returns framework-independent get, set, subscribe.
  • createBolt() returns React Provider, useApi, useSet, useStore.
  • useStore(path) normalizes the path, subscribes via subscribePathKey, reads via getByPathKey, and returns a path-bound setter.
  • listenersByPath is currently Map<string, Set<Listener>>. It is a render subscription index, not a data dependency graph.
  • set(path, valueOrUpdater) normalizes the path, writes the value, short-circuits no-op root identity, then calls notifyAll for root writes or notifyPrefixes for nested writes.
  • notifyPrefixes notifies root plus prefixes of the written path. For a.b.c, it notifies "", "a", "a.b", "a.b.c".
  • README.md currently says Bolt does not support derived/computed state. This feature changes that claim.
  • Package currently has no explicit test script in package.json. Add one if tests are introduced.
  • Current checkout has unrelated untracked bolt-mutation-analysis.html. Do not delete or stage it unless explicitly asked.

<h3>Manual State Policy</h3>
<ul>
  <li>Default: <code>manualWrites: "reject"</code>. A normal <code>set(targetPath, ...)</code> to a derived target throws. This prevents silent fights between user writes and derived writes.</li>
  <li>Escape hatch: <code>manualWrites: "allow"</code>. Normal writes can change the target, but the next source change may overwrite it. Document this as advanced behavior.</li>
  <li>Recommended dynamic UI pattern: manual state lives at a separate source path such as <code>B.override</code> or <code>B.local</code>; derived output lives at <code>B.value</code>.</li>
  <li>Do not support compute functions that call <code>set</code>. Compute returns one target value. Multi-target derivation can be a later feature.</li>
</ul>

Path Semantics

Derived dependencies must use affected-snapshot semantics, not current prefix-notification semantics.

Dependency Write Should recompute? Reason
A.x.y A.x.y Yes Exact source changed.
A.x A.x.y Yes Parent object snapshot changed by descendant write.
A.x.y A Yes Ancestor replacement may change descendant source.
A.x.y A.z No Sibling write cannot affect source snapshot.
"" Any path Yes Root dependency observes whole state.
Any path "" Yes Whole-store replacement may affect every source. 
function pathsOverlap(a: string, b: string) {
  if (a === "" || b === "") return true;
  return a === b || isPrefixPath(a, b) || isPrefixPath(b, a);
}


function isPrefixPath(prefix: string, path: string) {
  return path.startsWith(prefix + ".");
}

<h3>Why Not Only <code>sourcePath -&gt; nodes</code></h3>
<p>A source dependency on <code>A.x.y</code> must recompute when <code>A</code> is replaced. A source dependency on <code>A</code> must recompute when <code>A.x.y</code> changes. Exact source maps alone miss one side. Start with a simple scan across registered derived nodes, then add a trie later only if benchmarks require it.</p>

<h3>Hot Path Constraint</h3>
<ul>
  <li>If <code>derivedNodes.size === 0</code>, <code>set</code> should keep current behavior: write path, notify root/prefixes, return.</li>
  <li>If graph exists, call derivation settlement before notification.</li>
  <li>Do not run graph scans for empty graph.</li>
</ul>

<pre><code>settleDerived(initialChangedPathKeys)

<h3>Transitive Collection</h3>
<p>When <code>A</code> changes and <code>B</code> derives from <code>A</code>, <code>B</code> is affected. When <code>C</code> derives from <code>B</code>, <code>C</code> must also be included before notification. Build this set before recomputing so ordering can be stable.</p>
<pre><code>collectTransitiveAffectedNodes(changedPathKeys)

queue = [...changedPathKeys] affected = Set() seenChangeKeys = Set(queue)

while queue not empty: changed = queue.shift() for node of derivedNodes.values(): if node already affected: continue if any source overlaps changed: affected.add(node) if node.targetPathKey not in seenChangeKeys: seenChangeKeys.add(node.targetPathKey) queue.push(node.targetPathKey)

return affected

Topological Ordering And Cycles

Derived nodes form a graph. Node X must compute before node Y when Y has a source path overlapping X.targetPathKey.

edge X -> Y if pathsOverlap(X.targetPathKey, one of Y.sourcePathKeys)
  • Use DFS with visiting and visited sets.
  • If DFS reaches a visiting node, throw Bolt derived cycle: A -> B -> A.
  • Run cycle validation during derive() registration, and keep a defensive check during settlement.
  • Reject self-dependency: target path overlaps any source path. Example: target B.value, source B creates self-invalidating writes.
  • Allow sibling dependency: target B.value, source B.override is valid.

Notification Batching

Derived settlement must happen before React listeners fire. Otherwise components can render intermediate state: new A, stale B, stale C.

function notifyChangedPaths(changedPathKeys: Iterable<string>) {
  const listeners = new Set<Listener>();


  for (const pathKey of changedPathKeys) {
    if (pathKey === "") {
      collectAllListeners(listenersByPath, listeners);
    } else {
      collectPath(listeners, "");
      for (const prefix of pathPrefixes(pathKey)) {
        collectPath(listeners, prefix);
      }
    }
  }


  listeners.forEach((listener) => listener());
}
  • Dedup listeners so one component subscribed to multiple affected paths renders once.
  • Preserve current behavior for nested writes: root plus prefixes get notified.
  • Do not notify derived target subscribers until all derived paths settle.
  • Root replacement still notifies all subscribers.

README Contract

Add this conceptual contract in docs:

  • derive(target, sources, compute) materializes a target path.
  • Derived target acts like normal store state for reads and subscriptions.
  • Derived target updates before subscribers are notified.
  • Derived targets can chain.
  • Derived graph is optional; apps not using it pay no meaningful cost.
  • Derived compute must be synchronous and pure with respect to Bolt writes.
  • Use separate manual source paths for overrides or local state.
  • Cycles are invalid.

  • useApi().derive is available in React-created stores.
  • Source path writes recompute derived target before listener notification.
  • Derived target writes trigger downstream derived targets.
  • Chained derivation computes in dependency order.
  • Cycle registration throws.
  • Manual write to derived target rejects by default.
  • manualWrites: "allow" works and is tested.
  • Dispose removes graph registration.
  • Unrelated writes do not call compute.
  • Root writes recompute all affected derived targets.
  • Ancestor writes and descendant writes use overlap semantics.
  • Listeners are deduped across source and derived target notifications.
  • README documents the feature and removes outdated "no derived/computed state" limitation.
