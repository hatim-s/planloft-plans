AI App Builder — Progress & Roadmap

Updated July 17, 2026. The July 11 roadmap is superseded; status is verified against code on main, not commit messages.

The barebone builder has crossed the first useful product threshold: async data, richer blocks, a drag-and-drop canvas, session undo, and a substantially more steerable agent loop are all shipped.

Where we are

Shipped since July 11 — all merged to main, CI green.

Async-data slice 1 PRs #5–#10 · was R1

QueryRuntime is wired into the studio: reactive {$state} params, auto/manual modes, app-scoped queries, an SSRF-guarded CORS proxy, live Data-panel results, and runQuery/setState actions.

Table + Chart blocks was R3

Hardened with input validation, style-injection guards, and component tests.

Drag-and-drop canvas + Layers panel PRs #11–#12 · was R7

Palette drag-insert, reorder/reparent, and a hierarchy tree are all routed through undo tracking.

Session undo PRs #13–#20 · was R6

Op-log core with capture, inverse operations, preconditions, and a capped log; Cmd+Z, header Undo, granular and per-run agent reverts. Publishing intentionally clears the log; redo is out of v1 scope.

Agent steering & hardening R11 + R2 in tool form

SDK-native deliver/abort, a run-lifecycle state machine, debounced human-edit injection with per-page conflict tracking, multi-page awareness, 30-minute idle eviction, and actor/opId/runId attribution on every mutation.

Substrate + fix of the week

opSignals, per-block versions, replaceProps, and block-level writeSubtree isolation validation are in place. Agent bindings now target /queries/<id>/data/*, matching the runtime envelope rather than the prior incorrect /queries/<id>/* path.

Review signal

The R6/R11 stack landed through eight stacked PRs and a three-agent review fleet: 76 findings triaged, 34 fixed.

Next steps — priority × impact

Focus next on the public-product story while keeping independent UX and infrastructure work moving.

N1Serve published appsM

Finish R4: add the public /[appSlug] catch-all route, hydrate it from releases.getLive, use per-app slugs and release-pinned query configs. Publish becomes real.

The gap that blocks the product story: releases are snapshotted, but preview still renders draft blocks.

N2Clerk auth + ownershipM

Use ConvexProviderWithClerk and ctx.auth.getUserIdentity() in every function; enforce ownerId on all reads and writes.

Safety gate for everything public. It blocks N3 and makes N1 safe to expose.

N3Mutations + optimistic dataL

Implement the locked slice-2 design: MutationConfig, runMutation, invalidates, and append/update/remove optimistic verbs with automatic rollback. Shared type stubs already exist.

Read-only apps are demos; writes make them applications.

N4Interaction + bindings editor UXM

Replace raw JSON textareas with a catalog-validated event-to-action picker, a binding path picker with /queries/<id>/data and $state autocomplete, and a template helper.

The largest daily-use friction; unblocked today.

N5Cloud deploy + server-side secretsM

Deploy the agent service on Railway, move to Convex cloud, authenticate the agent HTTP API, and move query auth headers server-side.

Depends on N2. Query config credentials currently ship to every visitor, so public apps must not expose them.

N6Publish bolt to npmS

Publish @hatimcodes/bolt and replace the vendored package. Independent hygiene that unblocks external reuse.

N7Quality backlogS each

Batch eval rigor (N-iteration thresholds rather than a single smoke), redo as a post-v1 call, a DNS-rebind stance for the proxy, and structured agent-service logs keyed by sessionId/runId.

Lanes

Independent work can proceed now; the product-critical lane has a clear public-release sequence.

Lane A — Product-critical path N1 → N2 → N5

Publish → auth → deploy. This lane ends with a URL you can hand someone.

Lane B — Data N3

Slice-2 mutations; independent of Lane A.

Lane C — Editor UX N4

Independent, with a large perceived-quality win.

Lane D — Infra N6 + N7

Fill-in work that can run alongside the other lanes.

Recommended first wave

Start N1 + N4 + N6 in parallel today, with N2 immediately behind N1. That yields visitable published apps, a humane editor, and the path to a safe public deploy.
