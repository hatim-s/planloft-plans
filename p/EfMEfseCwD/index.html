AI App Builder — Progress & Roadmap

Barebone builder shipped — July 11, 2026. Human + agent co-editing the same page, live, end-to-end.

A builder where a human and an AI agent work on the same application simultaneously — pages stored as JSON specs, rendered to React at runtime, the agent a first-class collaborator editing through the same granular operations as the human.

Journey so far

From interrogation to a live co-editing loop in one sustained push.

Design grill June – July 2026

Three adversarial interview rounds (25 + 17 + 7 questions) stress-tested every macro choice. Output: ARCHITECTURE.md — 15 sections, a 44-entry decision log, and an explicitly ordered deferred list. Design declared locked before a line of product code was written.

Scaffold + contracts July 11

pnpm/turborepo monorepo scaffolded. docs/CONTRACTS.md carved the repo into three exclusive ownership zones with frozen shared interfaces (Convex function API, package exports, A↔C HTTP contract) and an append-only docs/INTEGRATION-NOTES.md channel. bolt vendored with PRs #3 (path-aware immutable writer) + #4 (derive) merged — 28 tests green. Convex local anon deployment + codegen wired.

Three parallel workstreams July 11

  • A — Builder UI (apps/studio, catalog registry): studio shell, 3-pane builder, chat panel, preview route, 7 shadcn-styled blocks + UnknownBlock fallback.
  • B — Backend & runtime (packages/backend, dsl, runtime): all contract Convex functions (20 tests), DSL parser / stream-parser / serializer with round-trip suite (10 tests), runtime stores + QueryRuntime + spec assembly (10 tests).
  • C — Agent service (services/agent): 8 tools, pi session wrapper, session manager, Hono HTTP server, ~200ms delta batcher (28 tests).
Process highlight

The build itself was multi-agent: Fable orchestrators ran the workstreams, Codex handled brute implementation, and Sonnet/Opus review passes caught issues pre-integration (finalize-retry bugs, prompt-rejection surfacing, typed-api gaps). The contract-and-ownership scheme let all three streams land without a single cross-stream file conflict.

Integration — two landmines found & defused July 11

  • Convex rejects $-prefixed field names — a platform document restriction that convex-test never surfaces. It broke the locked binding architecture end-to-end: json-render expressions ($bindState, $template) couldn't be stored or even passed as args. Fix: encodeDollarKeys/decodeDollarKeys codec in @app-builder/shared applied at the wire boundary — a single seam; A and C needed no structural changes.
  • Bare pi-agent-core breaks openai-codex OAuth — switched to pi-coding-agent's createAgentSession (the fallback path pre-approved in decision P1), restoring model access.
  • Plus: chat sessionId convention mismatch (A used pageId, C minted UUIDs) reconciled to sessionId = pageId; string function refs swapped for typed generated api.

Live smoke verified July 11

Full mutation-path smoke against the live local deployment passed (insert / setProps / setBindings / move / remove / publish / presence / messages; all studio routes 200). Then the real thing: a chat prompt produced a signup page — heading, bound email input, terms checkbox, submit button — appearing live on the canvas via agent → write_subtree DSL → Convex → subscription.

Where that leaves us

19 commits, 6 tested packages, 98 passing tests (backend 20 · agent 28 · bolt 28 · dsl 10 · runtime 10 · shared 2), typecheck green monorepo-wide, and a verified human+agent co-editing loop. The skeleton is real; everything below builds on it.

What works today

Verified against the repo and the July 11 live smoke. Dashes mark the honest gaps inside each path.

Human path

  • 3-pane builder at :3100 — palette / canvas / inspector
  • Click-insert from palette; selection + move/delete on canvas
  • Inspector prop editing (Zod-typed per catalog)
  • Preview route rendering the real spec — state bindings + on actions (validateForm) verified
  • Publish → immutable release snapshot + version pill
  • Optimistic granular mutations; fractional-index ordering; cycle-rejected reparent
  • Bindings edited as raw JSON textarea; no drag-and-drop; no undo

Agent path

  • Chat panel → agent service at :8787 (Hono; health / sessions / messages)
  • pi-coding-agent createAgentSession with openai-codex OAuth
  • All 8 contract tools: read_page · write_subtree · set_props · move_block · delete_block · list_pages · create_page · set_page_queries
  • DSL authoring: write_subtree parses HTML-like DSL → transactional Convex write
  • read_page returns DSL — agent reads what it writes (round-trip tested)
  • Presence pill (thinking / editing / idle); chat deltas batched ~200ms into Convex
  • Blocks appear at once per tool call — no streamed canvas growth yet

Infra

  • Convex local anon deployment; row-per-block schema per design
  • Read-then-patch granular mutations only (no tree replaces)
  • $-key codec at every wire boundary (@app-builder/shared)
  • releases.publish / getLive — frozen spec blobs + pointer flip
  • bolt 0.2.0-alpha vendored, PRs #3+#4 merged, 28 tests
  • 98 tests across 6 packages; every commit compiles
  • No auth, no cloud deployment, agent HTTP API unauthenticated (trusted-local only)

Next big leaps

From ARCHITECTURE.md §13's deferred list, integration notes, and known gaps. Sizes: S ≈ hours, M ≈ a focused day or two, L ≈ multi-day.

R1Wire QueryRuntime into the studioM

Page-query editor UI + /queries/{id} bindings + the runQuery custom action. The QueryRuntime class (TanStack QueryObserver → bolt page store) already exists and is tested — it's simply not mounted in the builder or preview.

Why it matters: this unlocks dynamic data apps — the core product promise. Until then the builder produces static pages.

Depends on: nothing — highest-leverage item on the board.

R2Streamed page-artifact authoringM

Agent emits <page-artifact> DSL as assistant text; the service SAX-parses it incrementally (createStreamParser already ships in @app-builder/dsl) and flushes batched inserts so the canvas grows live during generation. Today authoring is a single tool call — blocks appear all at once.

Why it matters: this is the signature demo moment and the design's chosen authoring pipeline (decision Q3).

Depends on: nothing hard; touches agent session plumbing, so sequence with R11.

R3Table + Charts blocksM

TanStack Table + shadcn markup; Recharts + shadcn ChartContainer. New catalog entries flow automatically into renderer, validation, and the agent's grammar (catalog-is-the-contract).

Why it matters: tables and charts are what internal-tool builders actually build. Deliberately deferred to P1-after-launch — that time is now.

Depends on: R1 — without real query data these blocks have nothing to show.

R4Published-app serving (public routes)M

Public catch-all route rendering releases by app slug via releases.getLive, adopting @json-render/next (SSR from frozen spec). Release snapshots already exist — there is no public renderer route yet, only the builder's preview.

Why it matters: "publish" currently produces a version number nobody can visit. This closes the build → ship loop.

Depends on: nothing for the route; R5 before any real public exposure.

R5Clerk auth + ownership enforcementM

ConvexProviderWithClerk, ctx.auth.getUserIdentity() in every function, ownerId enforced on all reads/writes. Single-owner, no orgs (decision V5).

Why it matters: hard prerequisite for any deployment beyond localhost; today every row is anonymous.

Depends on: nothing; blocks R12.

R6Session undo (Hatim's V4 design)L

In-memory capped client op log, humans only; agent ops arrive over the subscription tagged actor: agent and are individually revertible (inverse op, skip-if-overwritten-since); log nuked on publish. Undo executes as normal granular mutations.

Why it matters: granular revert of agent changes is the trust mechanism for co-editing — "the agent did something weird" must have a one-click answer.

Depends on: touches every mutation path — land after R7/R8/R9 settle the editor's write surface.

R7Drag-and-drop canvasM

Drag from palette + drag-to-reorder/reparent on canvas. Backend already speaks this language (moveBlock with beforeBlockId, fractional indexing, cycle rejection) — today's UI is click-insert + move buttons.

Why it matters: table-stakes builder ergonomics; the current interaction model reads as a prototype.

Depends on: nothing.

R8Interaction builder UIM

Visual editor for on: {event: {action, params}} — built-ins (setState, nav, validateForm) plus runQuery / navigate. The data model is already in place and verified (setBindings smoke-tested).

Why it matters: interactions are what separate an app from a mockup; today they're hand-written JSON.

Depends on: R1 for runQuery to be meaningful.

R9Bindings / expression editor UXS

Replace the inspector's raw JSON textarea with a purpose-built editor: path picker for $state/$bindState, template string helper, visible/repeat toggles.

Why it matters: bindings are the product's superpower and currently its worst UX; also removes a class of hand-typed $-key mistakes.

Depends on: nothing.

R10bolt npm publishS

Publish @hatimcodes/bolt 0.2.0 (PRs #3+#4 already merged in the vendored copy, 28 tests) and swap packages/bolt for the npm dependency.

Why it matters: removes vendored drift risk and gives bolt its own release cadence (decision S2).

Depends on: nothing; fully independent.

R11Agent hardeningM

Steering mid-generation (interrupt/redirect a running session), a deliberate read_page context-injection policy (when the agent sees human edits), and multi-page awareness beyond list_pages.

Why it matters: real co-editing sessions are messy — the human edits while the agent generates. Today's session is fire-and-forget per message.

Depends on: pairs naturally with R2 (same session internals).

R12Cloud deployment: Railway + Convex cloud + secrets proxyM

Agent service to Railway (decision V7), Convex from local anon to cloud, auth on the agent HTTP API (currently open + permissive CORS), then the credentialed-datasource Convex-action proxy (decision R3, deferred).

Why it matters: nothing is real until someone else can use it; the secrets proxy unlocks authenticated APIs — most real data sources.

Depends on: R5 (auth first). Secrets proxy also wants R1 in place.

Parallelization map

Five independent lanes — the same trick as the A/B/C barebone build. Arrows (→) are sequential within a lane; ∥ marks items that can run concurrently. All five lanes can start simultaneously.

Lane 1 — Data layer Workstream B profile

The core-promise lane. Runtime + backend heavy; strictly sequential — each step feeds the next.

R1 QueryRuntime wiringM→ R3 Table + ChartsM→ R12b Secrets/datasource proxyM

Lane 2 — Agent UX Workstream C profile

Same owner, same session internals. Streaming first (it restructures the session loop), hardening rides on the new shape.

R2 Streamed authoringM→ R11 Steering + context policyM

Lane 3 — Publish & auth Workstream A + infra profile

The "someone else can use it" lane. Public route can start immediately; auth gates real exposure; deploy last.

R4 Public release routesM→ R5 Clerk + ownershipM→ R12a Railway + Convex cloudM

Lane 4 — Editor UX Workstream A profile

Three parallel-safe UI builds (disjoint components), then undo — it instruments every mutation path, so it lands once the write surface stops moving.

R7 Drag-and-dropM∥ R9 Bindings editorS∥ R8 Interaction builderM→ R6 Session undoL

Lane 5 — Infra Anyone / anytime

Fully independent; slot in whenever.

R10 bolt npm publish + swapS
Cross-lane couplings (the only ones)
  • R8 (interactions) ← R1: the runQuery action needs QueryRuntime mounted. R8's UI shell can still be built against setState/nav first.
  • R3 (table/charts) spans ownership: catalog components are A-territory, data plumbing is B-territory — same coordination pattern as the barebone build, one integration note each way.
  • R2 ∥ human editing: streamed inserts + concurrent human edits is exactly the LWW/row-per-block design's home turf — no new machinery, but smoke-test the combination.
Recommended first wave

Kick off all five lanes at once with the barebone staffing pattern: R1 (Lane 1) + R2 (Lane 2) + R4 (Lane 3) + R7/R9 (Lane 4) + R10 (Lane 5). That wave alone turns the barebone into a demoable product: dynamic data, live-growing canvas, visitable published apps, and a builder that feels like one.
